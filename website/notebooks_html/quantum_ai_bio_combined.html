0. Theoretical Foundation: Why Hybrid Quantum-Classical? ¶

The big idea in plain language¶

A quantum computer is not replacing the classical computer in this notebook. Instead, the two play different roles:

  • The classical computer keeps track of data, chooses parameters, and evaluates progress.
  • The quantum routine provides a different way to represent and explore candidate solutions.

That is why this is called a hybrid workflow.

Why not use only a quantum computer?¶

Today’s quantum devices are still limited. This period is often called the NISQ era: noisy intermediate-scale quantum computing.

In practice, that means:

  • circuits must stay fairly short,
  • measurements are noisy,
  • and classical hardware is still better for most of the surrounding optimisation work.

So a realistic beginner mental model is:

The classical computer is the manager. The quantum circuit is a specialised worker used for one part of the search.

The feedback loop¶

         ┌─────────────────────────────────┐
         │  Classical computer             │
         │  chooses QAOA parameters        │
         │  and decides what to try next   │
         └──────────────┬──────────────────┘
                        │  send (gamma, beta)
         ┌──────────────┼──────────────────┐
         │  Quantum routine                │
         │  prepares a circuit,            │
         │  evaluates the candidate        │
         │  and returns a score            │
         └──────────────┬──────────────────┘
                        │  return expected cut
                        └──> update parameters

What QAOA is doing here¶

QAOA stands for Quantum Approximate Optimization Algorithm.

For this notebook, you can think of it as a tunable recipe:

  • start from a simple quantum state,
  • apply one set of operations related to the problem,
  • apply another set of operations that encourage exploration,
  • measure how good the result is,
  • then adjust the recipe settings and try again.

Those recipe settings are the angles $\gamma$ and $\beta$.

The mathematical view¶

The optimisation target in variational quantum algorithms is usually written as

$$\mathcal{F}(\boldsymbol\theta) = \langle\psi(\boldsymbol\theta)|\hat{H}|\psi(\boldsymbol\theta)\rangle$$

If that expression is unfamiliar, the only idea you need is this:

  • $\boldsymbol\theta$ are tunable parameters,
  • $|\psi(\boldsymbol\theta)\rangle$ is the quantum state created by those parameters,
  • $\hat{H}$ is the quantity we care about,
  • and the expression returns a score.

For QAOA in this notebook, we tune $(\gamma, \beta)$ to get as large an expected cut value as possible.

Why bring in a GNN?¶

Running a fresh search for good QAOA parameters on every new graph can be slow. A Graph Neural Network helps by learning a shortcut:

  • input: the graph structure,
  • output: a good guess for $(\gamma, \beta)$.

In other words, instead of saying:

"Let us optimise from scratch for every graph,"

we try:

"Let us learn the pattern of good settings across many graphs."

Why graphs matter on both sides¶

This notebook uses graphs twice:

  1. In the QAOA example, the graph is the optimisation problem itself.
  2. In the biomedical example, the graph connects similar patients.

That is the core unifying theme of the notebook: graph structure is the shared language between the quantum demo and the AI demo.

Part 1: QAOA - Solving a Small Graph Optimisation Problem ¶

The problem: MaxCut¶

We start with a graph and ask a simple question:

Can we split the nodes into two groups so that as many edges as possible go across the split?

That problem is called MaxCut.

A non-technical way to picture it is this: if edges represent relationships, we want a split that separates as many connected pairs as possible.

Why MaxCut is useful in tutorials¶

MaxCut is popular in quantum optimisation examples because:

  • it is easy to describe,
  • it maps naturally to binary decisions,
  • and it is still rich enough to show how QAOA works.

The quantum formulation¶

In QAOA, the optimisation problem is converted into a score operator called the cost Hamiltonian:

$$\hat{C} = \frac{1}{2}\sum_{(u,v)\in E}(I - Z_u Z_v)$$

If you are not used to quantum notation, do not get stuck here. The important interpretation is:

  • every edge contributes to the score,
  • edges that are successfully separated increase the cut value,
  • and the quantum state is evaluated according to that score.

The two QAOA ingredients¶

QAOA alternates between two types of operations:

  1. Cost step This pushes the quantum state to reflect the structure of the MaxCut objective.
  2. Mixer step This prevents the search from getting stuck too early and keeps multiple possibilities in play.

At depth $p=1$, there is only one cost angle and one mixer angle:

  • $\gamma$ controls the problem-dependent step,
  • $\beta$ controls the exploration step.

The state we optimise¶

The variational state is written as

$$|\gamma, \beta\rangle = U_B(\beta)\,U_C(\gamma)\,|+\rangle^{\otimes n}$$

For beginners, the practical meaning is simple:

  • start from an equal starting point,
  • apply a problem-aware transformation,
  • apply a mixing transformation,
  • then score the result.

What the optimisation loop is trying to do¶

We want to maximise the expected cut value:

$$\langle C\rangle(\gamma, \beta) = \langle\gamma,\beta|\hat{C}|\gamma,\beta\rangle$$

You can read this as:

"For this choice of angles, how good is the average solution quality?"

Why we also compute the exact answer¶

Because the demo graph is small, we can still compute the true best MaxCut by checking all possible partitions. That gives us a reference point:

  • exact MaxCut = the true best answer,
  • QAOA expected cut = the score achieved by the chosen QAOA parameters.

This makes it easy to judge whether QAOA is doing well.

What to watch for in the next code cell¶

The code below does three things:

  1. builds a small random graph,
  2. finds the exact best cut by brute force,
  3. runs a classical search to find strong QAOA angles for that graph.

That classical search gives us the baseline the GNN will later try to imitate.

# ─────────────────────────────────────────────────────────────────────────────
# CELL 1: Generate MaxCut graph + classical QAOA optimisation
# ─────────────────────────────────────────────────────────────────────────────

import time, math

p = 1   # QAOA depth — only 2 variational parameters (γ₁, β₁)
n = 6   # qubits / graph vertices  (2^6 = 64 amplitudes)

# ── Sample Erdős–Rényi random graph G(n, p_edge=0.5, seed=7)
# E[m] = p_edge * C(n,2) = 0.5 * 15 = 7.5 edges
G_nx  = sample_erdos_renyi(n, p_edge=0.5, seed=7)
edges = list(G_nx.edges())

print(f"MaxCut graph: n={n}, m={len(edges)}")
print(f"Edges: {edges}")
print(f"Degree sequence: {sorted([d for _,d in G_nx.degree()], reverse=True)}")
print(f"Is connected: {nx.is_connected(G_nx)}")

# ── Brute-force MaxCut (exact, O(2^n * m))
best_cut = 0
best_partition = 0
for mask in range(1, 2**n):
    cut = sum(1 for (u,v) in edges
              if ((mask >> u) & 1) != ((mask >> v) & 1))
    if cut > best_cut:
        best_cut = cut
        best_partition = mask
print(f"\nExact MaxCut  : {best_cut}")
print(f"Best partition: {format(best_partition, f'0{n}b')}  "
      f"(S={[i for i in range(n) if (best_partition>>i)&1]}, "
      f"S̄={[i for i in range(n) if not (best_partition>>i)&1]})")

# ── Classical QAOA optimisation: minimise -⟨C⟩(γ, β) via Nelder–Mead
def classical_optimize(n, edges, p, maxiter=300, seed=0):
    """
    Nelder–Mead minimisation of the negative QAOA expected cut.

    The parameter space is [0, π]^p × [0, π/2]^p.
    At p=1, this is a 2D unconstrained optimisation.
    """
    rng = np.random.default_rng(seed)

    def neg_exp(x):
        g, b = x[:p], x[p:]
        return -expected_cut(n, edges, qaoa_state(n, edges, g, b))

    x0 = rng.uniform(0, math.pi, size=2 * p)

    res = minimize(neg_exp, x0, method='Nelder-Mead',
                   options={'maxiter': maxiter, 'xatol': 1e-5, 'fatol': 1e-5})
    return res.x[:p], res.x[p:], -res.fun

t0 = time.perf_counter()
opt_g, opt_b, val_opt = classical_optimize(n, edges, p)
classical_time = time.perf_counter() - t0

print(f"\nNEQA Classical Optimisation (Nelder–Mead, p={p})")
print(f"  γ_opt   = {opt_g}")
print(f"  β_opt   = {opt_b}")
print(f"  ⟨C⟩_opt = {val_opt:.4f}  (exact MaxCut = {best_cut})")
print(f"  Approx ratio: {val_opt/best_cut:.4f}")
print(f"  Time    = {classical_time*1000:.2f} ms")

# ── Save artifacts
pd.DataFrame(edges, columns=['u', 'v']).to_csv(
    os.path.join(proj_root, 'outputs', 'maxcut_graph.csv'), index=False)
pd.DataFrame({'opt_gamma': opt_g, 'opt_beta': opt_b}).to_csv(
    os.path.join(proj_root, 'outputs', 'qaoa_classical_angles.csv'), index=False)
print(f"\nSaved: outputs/maxcut_graph.csv, outputs/qaoa_classical_angles.csv")

Part 2: GNN Parameter Prediction ¶

The bottleneck we are trying to remove¶

In Part 1, the classical optimiser had to search for good QAOA angles for one graph. If you repeat that process for many graphs, it becomes expensive.

So the question becomes:

Can we train a model to look at a graph and directly guess good QAOA settings?

That is the purpose of the GNN in this section.

Intuition before the math¶

Graphs with similar structure often need similar QAOA parameters. A graph neural network is a natural model for learning that pattern because it is built to read graph structure.

In plain language:

  • the optimiser in Part 1 learns one graph at a time,
  • the GNN tries to learn the general rule across many graphs.

Why this matters¶

If the GNN works well, then for a new graph we can do this:

  • input the graph,
  • run one forward pass,
  • get predicted values for $\gamma$ and $\beta$.

That is much faster than a fresh search loop.

What the model sees¶

This notebook uses a lightweight GCN-style model. Its inputs are deliberately simple:

  • the graph adjacency structure,
  • and a node feature based mainly on degree.

That keeps the demo understandable while still showing the full hybrid idea.

The core update rule¶

A graph convolution layer can be written as

$$\mathbf{H}^{(l+1)} = \text{ReLU}\!\left(\hat{D}^{-1/2}\hat{A}\hat{D}^{-1/2}\,\mathbf{H}^{(l)}\,\mathbf{W}^{(l)}\right)$$

If you are new to this notation, here is the practical meaning:

  • each node updates its representation,
  • information is mixed with nearby nodes,
  • and the network gradually builds a summary of the graph.

After a few such layers, we pool the node representations into one graph-level vector and use that to predict the QAOA angles.

What the output means¶

The GNN does not solve MaxCut directly. It predicts the settings for the QAOA circuit.

So the flow is:

  1. graph goes into the GNN,
  2. GNN predicts $(\hat\gamma, \hat\beta)$,
  3. those parameters are fed into QAOA,
  4. QAOA produces an expected cut score.

Why simple node features can still work¶

Degree is a basic structural signal, but it already tells us a surprising amount:

  • which nodes are hubs,
  • how uneven the graph is,
  • and roughly how connected the instance is.

For a tutorial, this is a good tradeoff: the feature design stays simple while the hybrid pipeline remains realistic.

What to watch for in the next code cells¶

The training cell teaches the GNN to output useful QAOA angles. Then we compare:

  • the angles found by classical search,
  • the angles predicted by the GNN,
  • and the expected cut scores produced by both.

Part 3: Expected-Cut Landscape and Parameter Heatmap ¶

Why visualise the landscape?¶

Once QAOA has parameters, a natural question is:

What happens if we change them?

The expected cut value $\langle C\rangle(\gamma, \beta)$ gives a score for each angle pair. If we plot those scores, we get a landscape.

A beginner-friendly interpretation is:

  • high regions = better QAOA settings,
  • low regions = worse QAOA settings,
  • peaks = promising choices for the optimiser.

Why this plot is useful¶

The landscape helps answer several practical questions:

  • Is the optimisation problem smooth or messy?
  • Is there one clear good region or many competing ones?
  • Does the GNN prediction land near a strong region, even if it is not exactly perfect?

What the axes mean¶

  • $\gamma$ controls how strongly the circuit reflects the MaxCut objective.
  • $\beta$ controls how strongly the circuit mixes between possibilities.

Together they determine the QAOA behaviour for this depth-$p=1$ example.

A note on the mathematics¶

For tiny graphs, some QAOA expressions can be written in closed form. For the more general case in this notebook, we use a numerical simulator instead. That is easier to understand and closer to how the rest of the pipeline is implemented.

What counts as a good GNN prediction?¶

A strong prediction does not need to hit the exact best point. It only needs to land close enough to a high-value area that the resulting score stays competitive.

That matters because in practice we often care more about:

  • getting a good answer quickly,
  • than finding the mathematically perfect angle pair every time.

Right - the full two-dimensional landscape
This heatmap shows the expected cut score across many $(\gamma, \beta)$ combinations.

A simple way to interpret the colors is:

  • brighter areas = better QAOA settings,
  • darker areas = worse settings,
  • markers show where the classical optimiser and the GNN prediction land.

What you should look for¶

  • If the bright region is wide, the problem is more forgiving.
  • If the bright region is narrow, small parameter mistakes matter more.
  • If the GNN lands near the bright region, it has learned something useful even if it is not exactly on the peak.

For this notebook’s small $p=1$ example, the landscape is usually smooth enough that both optimisation and interpretation remain manageable.

Part 4: Biomedical GCN - Graph-Structured Cancer Diagnosis ¶

What changes in this section?¶

The quantum example used a graph as an optimisation problem. Here, we use a graph in a different way: to connect patients who look similar based on their medical measurements.

Dataset: Wisconsin Diagnostic Breast Cancer (WDBC)¶

  • Samples: 569 patients
  • Features: 30 numeric measurements extracted from biopsy images
  • Classes: malignant or benign
  • Use in this notebook: a compact, well-known benchmark for medical classification

Why turn patient data into a graph?¶

A standard neural network treats each patient mostly on their own. A graph model adds one extra idea:

patients who are similar can help inform each other’s representation.

That can be useful when borderline cases sit near clearer cases in feature space.

What the GCN is doing¶

The model builds a patient-similarity graph using k-nearest neighbours. Then each layer updates a patient representation by mixing:

  • the patient’s own features,
  • and information from nearby similar patients.

The equations are included below for completeness, but the practical story is:

  • start with raw clinical measurements,
  • pass messages through the patient graph,
  • output a predicted class for each patient.

$$\text{Input: }\mathbf{X} \in \mathbb{R}^{n\times 30}, \quad \hat{A} \in \mathbb{R}^{n\times n}$$

$$\mathbf{H}^{(1)} = \text{ReLU}(\hat{A}_{\rm norm}\,\mathbf{X}\,\mathbf{W}^{(0)})$$

$$\mathbf{H}^{(2)} = \text{ReLU}(\hat{A}_{\rm norm}\,\mathbf{H}^{(1)}\,\mathbf{W}^{(1)})$$

$$\hat{y}_i = \text{softmax}\!\left(\mathbf{H}^{(2)}_i\,\mathbf{W}^{(2)}\right)$$

Why this section belongs in the same notebook¶

At first glance, MaxCut and cancer classification look unrelated. The common idea is actually simple:

  • both problems are represented as graphs,
  • both use graph neural networks,
  • and both show how structure can improve decision-making.

The quantum part focuses on optimisation. The biomedical part focuses on prediction. Together they show two different ways graph-based AI can be useful.

class BioGCN(nn.Module):
    """
    Two-layer GCN for transductive node classification.

    Forward pass:
        H¹ = ReLU(A_norm · X · W⁰)         shape (n, hidden)
        H² = ReLU(A_norm · H¹ · W¹)        shape (n, hidden//2)
        logits = H² · W²                   shape (n, n_classes)

    Note: bias=False follows Kipf & Welling (2017) convention,
    which works well with batch normalisation but is also common without it.

    Args:
        in_dim    : input feature dimension (30 for WDBC)
        hidden    : first hidden layer dimension
        n_classes : number of output classes (2)
    """
    def __init__(self, in_dim: int, hidden: int, n_classes: int):
        super().__init__()
        self.fc1 = nn.Linear(in_dim,      hidden,    bias=False)
        self.fc2 = nn.Linear(hidden,      hidden//2, bias=False)
        self.clf = nn.Linear(hidden//2,   n_classes)
        # Optional: add dropout for regularisation
        self.drop = nn.Dropout(p=0.1)

    def forward(self, A: torch.Tensor, X: torch.Tensor) -> torch.Tensor:
        h = F.relu(A @ self.fc1(X))      # Layer 1: graph conv + ReLU
        h = self.drop(h)
        h = F.relu(A @ self.fc2(h))      # Layer 2: graph conv + ReLU
        return self.clf(h)               # Classifier head (logits)

# ── Convert to tensors
A_t = torch.tensor(A_norm, dtype=torch.float32)
X_t = torch.tensor(X_std,  dtype=torch.float32)
y_t = torch.tensor(y,       dtype=torch.long)

# ── Train/test split (stratified 80/20)
idx               = np.arange(len(y))
tr_idx, te_idx    = train_test_split(idx, test_size=0.20, stratify=y, random_state=42)
tr_mask           = torch.zeros(len(y), dtype=torch.bool); tr_mask[tr_idx] = True
te_mask           = torch.zeros(len(y), dtype=torch.bool); te_mask[te_idx] = True

print(f"Train samples: {tr_mask.sum().item():3d}  "
      f"(mal: {y[tr_idx][y[tr_idx]==0].shape[0]}, "

      f"ben: {y[te_idx][y[te_idx]==1].shape[0]})")

# ── Model + optimiser

# ── Training loop

print(f"\nTraining BioGCN for {epochs} epochs...")
print("-" * 55)

for epoch in range(1, epochs + 1):
    bio_model.train()
    optimizer.zero_grad()
    logits = bio_model(A_t, X_t)

    optimizer.step()

    history['loss'].append(loss.item())

              f"train_acc={tr_acc:.4f}  |  test_acc={te_acc:.4f}")

print("-" * 55)

Synthesis: How QAOA and GNN Connect ¶

The shortest possible summary¶

This notebook tells one story in two settings:

  • in the quantum optimisation section, a GNN helps choose good QAOA parameters for a graph problem,
  • in the biomedical section, a GCN predicts patient labels on a similarity graph.

So the shared theme is not "quantum everywhere." The shared theme is:

graphs provide structure, and learning methods can use that structure in different ways.

Side-by-side view¶

Component What the graph represents What the model outputs Main goal
QAOA + GNN a MaxCut optimisation instance predicted QAOA angles $(\hat\gamma, \hat\beta)$ speed up parameter selection
Biomedical GCN a similarity network of patients class predictions improve diagnosis support

Why this hybrid idea is interesting¶

The QAOA section shows a realistic hybrid pattern:

  • a quantum-style routine can evaluate an optimisation objective,
  • but a classical learned model can help remove repeated search overhead.

That is important because many conversations about quantum computing stay too abstract. This notebook makes the workflow concrete:

  • define a graph problem,
  • run QAOA on it,
  • train a GNN to predict useful parameters,
  • compare quality and speed.

How to explain this to a non-technical audience¶

A recruiter-friendly version would be:

"This project explores how graph-based AI can support quantum optimisation by learning good starting settings, while also demonstrating graph learning on a biomedical classification task."

That description is accurate without overselling the current state of quantum hardware.
